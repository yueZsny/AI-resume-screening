# Git 提交规范

## 提交后记录（每次 commit，强制）

每次执行 `git commit` 后，必须新增一条提交记录；未记录视为提交流程未完成。

### 1) 记录位置

- 固定目录：`docs/releases/`
- 不得写到其它目录（如 `md/releases/`、根目录等）

### 2) 文件命名规范

- 文件名必须包含“提交时间 + 分支名”
- 推荐格式（精确到秒，避免重名）：
  - `YYYY-M-D HH-mm-ss <branch>.md`
  - 示例：`2025-12-5 14-23-44 dev-v1.md`
- 如只写日期（如 `2025-12-5 dev-v1.md`），同一天多次提交会冲突，不建议使用

### 3) 记录粒度（禁止合并）

- 一次提交 = 一份记录文件
- 严禁把多次 commit 写进同一个文件
- 允许同一天同一分支存在多份记录（按提交时间区分）

### 4) 记录时机

- 必须在本次 commit 完成后立即补记录
- 在下一次 commit 或 push 前，先补齐上一条记录

### 5) 最低必填内容

每个记录文件至少包含以下字段：

1. 提交时间（含时区）
2. 分支名
3. Commit hash（完整或短 hash）与 Commit message
4. 影响文件列表（逐条列出）
5. 改动说明（做了什么、影响了什么）

### 6) 推荐补充内容（建议）

- 提交类型：`feat / fix / chore / docs / refactor / test`
- 变更原因（为什么做）
- 风险点（可能影响哪里）
- 回滚方式（如 `git revert <hash>`）
- 验证方式（执行的命令和结果）

### 7) 记录模板（可直接复制）

```md
# 提交记录

- 提交时间：2025-12-5 14:23:44 +0800
- 分支：`dev-v1`
- Commit：`abc1234`
- Message：`feat(xxx): ...`

## 影响文件

- `path/a.ts`
- `path/b.tsx`

## 改动说明

- 做了什么：
- 影响什么：
- 验证结果：
```

### 8) 自检清单（提交前）

- [ ] `docs/releases/` 下已新增本次 commit 对应文件
- [ ] 文件名包含提交时间和分支名
- [ ] 文件内容包含最低必填字段
- [ ] 未把多次 commit 合并到同一个记录文件
