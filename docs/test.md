# 测试规范

## 测试技术栈

- 使用 `Playwright` 作为端到端（E2E）测试工具。

## 强制要求

1. 每个新功能都必须编写对应的 Playwright E2E 测试。
2. 测试目标是回归验证（Regression Testing），避免已有功能被新改动破坏。
3. 每次任务完成后，必须执行相关 Playwright 测试。

## 真实链路要求

1. 测试应优先覆盖真实业务链路，而不是只验证前端交互流程。
2. 用例需尽量走通“页面操作 -> 接口请求 -> 服务处理 -> 数据持久化/读取 -> 页面结果”的完整路径。
3. 核心流程（如登录、创建、更新、删除、发布、下载等）禁止仅靠前端 mock 判定通过。
4. 若因外部依赖限制必须使用 mock，需在测试说明中明确 mock 范围、原因与未覆盖风险。
5. 默认禁止在 E2E 中对项目内部 API（`/api/**`）使用 `page.route` 进行拦截模拟。
6. 新增或重写用例时，必须提供真实链路版本；仅前端交互版不视为完成。

## 用例去重与步骤复用规范

1. 新增用例前，必须先检查现有 `tests/**/*.spec.ts`，避免对同一业务场景重复覆盖。
2. 每条用例必须有唯一测试目标（一个核心行为 + 一个核心断言）；目标重复的用例应合并，不得平行新增。
3. 若两个用例的前置条件、操作步骤、期望结果高度一致（仅文案/数据名不同），视为重复用例，必须改为参数化或删除冗余项。
4. 可复用步骤（登录、初始化数据、清理数据、公共导航）必须抽到 `tests/helpers/**` 或 `beforeEach`，单个用例中仅保留与本场景差异相关的步骤。
5. 同一模块只保留 1 条主干 Happy Path；其余用例应覆盖边界、异常、权限、回归风险点，禁止重复验证同一主流程。
6. 提交测试改动时，需在说明中写明“本用例与已有用例的差异点”，无法说明差异的用例不应提交。
